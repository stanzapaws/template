# For Agents

## Writing Documentation

Write like a careful, knowledgeable human.

- Be specific. Use concrete facts; avoid statements that could apply broadly.
- Do not inflate importance. Avoid words like “crucial,” “pivotal,” or claims about broader impact unless supported.
- Cut all superficial analysis. No “highlighting,” “underscoring,” or implied meaning.
- Use plain language. Prefer “is/has” over “serves as/boasts/offers.” No promotional tone.
- Avoid AI-style patterns: buzzwords, filler phrasing, “not just X but Y.”
- Do not generalize or vaguely attribute (“experts say,” “studies show”) without specifics.
- Keep structure natural. No forced conclusions or template sections.
- Be concise. Remove anything that doesn’t add information.
- Do not use emojis.

Final check: If a sentence could fit many unrelated topics, rewrite or delete it.

## Writing JavaScript

Write JavaScript like an experienced developer maintaining a real codebase.

- Solve the problem directly. Avoid over-engineering or unnecessary abstraction.
- Prefer simple, readable code over clever or flexible designs.
- Do not add features, options, or configurability unless required.
- Avoid unnecessary defensive code unless there is a clear failure mode.
- Keep functions focused; don’t split logic without a clear benefit.

Style:

- Use clear, conventional names.
- Prefer straightforward control flow.
- Comment only non-obvious decisions. No tutorial-style comments.

Avoid:

- Unnecessary classes, patterns, or abstraction layers.
- Wrapping built-in functionality without need.
- “Future-proofing” without a real use case.
- Unnecessary async/await.

Output:

- Write code suitable for immediate production use.
- Choose the simplest reasonable solution when uncertain.

Final check:

- Remove anything overbuilt, generic, or unused.
- If it wouldn’t pass a real code review, simplify or delete it.
